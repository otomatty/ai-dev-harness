# レポート修正の振り分け(report-revision)

生成済みレポートへの修正依頼を分類し、コストと能力の釣り合う担当に委譲する。

## 対象

research-writeup が出力した成果物一式:
- レポート `*.html`
- 元データ `data.json`(可能な限りこちらを直して再生成する)
- `render.py`(スタイル・図解の定義)

修正は原則 **data.json を直して render.py で再生成** する。スタイル・図解自体を変える場合のみ render.py を編集する。

## 分類と委譲先

依頼を読み、次のどれかに振り分ける。

### 1. 軽い修正 → report-copyeditor(安いモデル / haiku)
文言・誤字・言い回し・トーン・簡潔化・語順・ラベル表記など、**意味や事実を変えない**テキスト修正。
- 例:「ここの表現をもっと簡潔に」「敬体に統一」「誤字を直して」「見出しの文言を変えて」
- data.json の `claim` / `interpretation` / `options_note` などの**文字列だけ**を編集。
- 触ってはいけない:`citation`(原文・訳・URL・版)、`badge`、数値、構造。事実は変えない。

### 2. 重い修正 → report-reviser(少し性能の高いモデル / sonnet)
図解の改善・スタイル/配色/レイアウト変更・セクションの追加削除や再構成・表構造の変更・複数箇所にまたがる書き換え。
- 例:「ポジショニング図の軸を変えて」「配色を落ち着いた色に」「比較表を候補ごとの節に組み替えて」「意思決定フローを分かりやすく」
- 図解・スタイルは `render.py` の CSS / SVG関数を編集(全レポートに反映)。構成変更は data.json の構造を編集。
- 事実・原文引用は保持する。判断が要る書き換えでも、根拠(citation)は消さない。

### 3. 事実の追加調査が必要 → revision ではなく深掘りへ
「なぜ」「もっと調べて」「この数値の裏付けは?」など、**新しい情報の確認が必要**な依頼は、report-revision で文章をいじらず、tech-selection-research 経由で **candidate-researcher の深掘りモード**に回す。revision エージェントは事実を創作してはならない。

## 手順

1. 依頼を上の3分類に振り分ける(複数該当なら分割して各担当へ)。
2. 対象ファイル(data.json / render.py / html)のパスを担当に渡す。どこをどう直すかを具体的に指定する。
3. 担当が編集 → render.py で再生成 → 差分の要点を報告。
4. 再生成後のHTMLをユーザーが確認できるようにする。

## 守ること

- 事実と原文引用は revision で創作・改変しない(誤字修正を除く)。要出典の変更は深掘りに回す。
- 修正は対象箇所に限定し、無関係な部分を書き換えない。
- 生成物は上書きせず、必要に応じてバージョン付きで残す。
