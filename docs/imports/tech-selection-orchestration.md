# 技術選定リサーチ運用ルール(抜粋)

技術選定・ライブラリ比較・「どれを採用すべきか」の相談では、**tech-selection-research** スキルを使う。

## 調査の委譲(本線を汚さない)
各候補の調査は **candidate-researcher** サブエージェント(Task)へ委譲する。
- 候補ごとに1 Task(独立候補は並列)。渡すのは最小限:候補名 / 評価軸(重み) / must・want / JSON断片の出力パス。本線の全文脈は渡さない。
- サブエージェントは一次情報を**原文引用**しながら裏取りし、`findings` 用のJSON断片(主張+原文引用+バッジ)を書き出す。本線には要約とパスだけを返す。
- 本線は断片をマージして data.json を作り、生の検索結果は貼らない。

## 成果物(原文引用つきHTML)
`python .claude/skills/research-writeup/assets/render.py <data.json> <out.html>`
- 調査ごとに新しいHTMLを作る(上書きしない)。
- 解説で引用したら、その直後に「原文+日本語訳(非日本語のみ)+リンク」を置く。**末尾に出典を一括しない。** 引用は最小限。
- 事実と解釈を視覚分離し、裏取りをバッジ(確認済/要確認/不明)で示す。
- **レーダーチャートは使わない。** 図解はポジショニング(任意)と意思決定フローのみ。
- スタイル/図解を変えるときは render.py 上部の CSS と SVG関数だけを編集。

## フォローアップの振り分け
- **「なぜ」「もっと調べて」「裏付けは?」** → 該当候補を特定し、candidate-researcher を**深掘りモード**で再委譲。返った items を findings にマージして再生成。
- **「内容を修正したい」** → **report-revision** スキルへ。
  - 文言・誤字・トーン・簡潔化など軽い修正 → **report-copyeditor**(haiku、安い)
  - 図解・レイアウト・配色・構成など重い修正 → **report-reviser**(sonnet、高め)
  - 事実の追加確認が要る修正は revision ではなく深掘りへ戻す。

## 裏取り
外部情報を根拠にするときは **source-verification** の階層と日付/バージョン確認に従い、[確認済/要確認/不明] で区別する。原文が示せない主張は確定情報にしない。
